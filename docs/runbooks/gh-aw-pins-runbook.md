# gh-aw Pins & Lock Files Runbook — Keeping Agentic Workflows Reproducible and Lint-Clean

This runbook covers operating **GitHub Agentic Workflows (`gh-aw`)** in any
`modeled-information-format` repo: how the generated files are pinned, when and
how to refresh pins (`--force-refresh-action-pins`, `actions-lock.json`), and how
to keep the org's standalone linters (actionlint, checkov, pin-check) green on
gh-aw's generated output. The worked example throughout is `MIF`'s
`daily-docs-review` workflow.

Mechanics it builds on (do not duplicate here):

- Org branch-protection / required checks: `docs/runbooks/branch-protection-runbook.md`.
- The reusable lint/scan workflows whose jobs become required checks live in this
  repo's `.github/workflows/` (actionlint, checkov, pin-check, sast, …).

## Golden rules

1. **`.md` is the source; `.lock.yml` is immutable build output.** Never hand-edit
   a `.lock.yml` (or any file headed `# … DO NOT EDIT`, e.g. `agentics-maintenance.yml`).
   Edit the `.md` and recompile.
2. **Validate with gh-aw's own commands**, not a hand-rolled diff: `gh aw validate`,
   `gh aw lint`, `gh aw compile`.
3. **Recompile is authoritative for pins.** Dependabot bumping a SHA in a generated
   `.lock.yml` without recompiling causes drift (see [Drift](#drift-dependabot-vs-recompile-101)).
4. **The org runs *standalone* actionlint/checkov** that don't know gh-aw's
   extensions. gh-aw's generated quirks are scoped-ignored, not hand-fixed.
5. **Check `gh aw --help` before asserting how a flag behaves.** Do not guess
   gh-aw internals.

## The files and how they're pinned

| File | What it is | Mutability | Pinning |
| --- | --- | --- | --- |
| `.github/workflows/<name>.md` | Workflow **source** (frontmatter + agent prompt) | Hand-authored | — |
| `.github/workflows/<name>.lock.yml` | Compiled GitHub Actions workflow | **Immutable, generated** (`gh aw compile`) | Actions SHA-pinned; **container images digest-pinned inline** (`ghcr.io/…:tag@sha256:…`) |
| `.github/workflows/agentics-maintenance.yml` | Generated maintenance workflow (`# … DO NOT EDIT`) | **Immutable, generated** | as above |
| `.github/workflows/copilot-setup-steps.yml` | Scaffolded **template** (no DO-NOT-EDIT marker) | Hand-editable | You SHA-pin its `uses:` yourself (see [pin-check](#pin-check-scaffold-files)) |
| `.github/aw/actions-lock.json` | Pin **cache** for gh-aw action refs (and, on older stacks, container digests) | Generated | Refreshed by `--force-refresh-action-pins` |
| `.github/{agents,skills,aw}/` | gh-aw agent/skill/prompt scaffolding | Generated by `gh aw upgrade` | — |

Strict mode is **on by default** (`"strict":true` in the lock header) and enforces
action pinning, network config, and safe-outputs; it refuses write permissions and
deprecated fields. Leave it on.

## Command reference (verified via `gh aw --help`)

| Command | Does | Emits `.lock.yml`? |
| --- | --- | --- |
| `gh aw compile` | Compile `.md` → `.lock.yml` | yes |
| `gh aw compile --force-refresh-action-pins` | Clear the pin cache and re-resolve all action SHAs from the GitHub API; rewrites `actions-lock.json` | yes |
| `gh aw validate` | `compile --validate --no-emit` + all linters (zizmor/actionlint/poutine). The right pre-push check — **no working-tree changes** | no |
| `gh aw lint [file\|dir]` | actionlint on existing `.lock.yml` (no recompile) | no |
| `gh aw upgrade` | Update the extension + agent files, apply `.md` codemods, recompile, scaffold `.github/{agents,skills,aw}` | yes |
| `gh aw update` | 3-way-merge a workflow from its source repo + recompile | yes |

Pinning gh-aw's own action refs: `--action-mode release|action` + `--action-tag <sha|tag>`,
or `--gh-aw-ref <branch|tag|sha>` (resolves to an immutable SHA at compile time).
Avoid `--allow-action-refs` (permits unresolved refs).

## Procedures

### Routine: edit a workflow

1. Edit `.github/workflows/<name>.md` only.
2. `gh aw compile` → regenerates `<name>.lock.yml`.
3. `gh aw validate` → 0 errors.
4. Commit **both** the `.md` and the regenerated `.lock.yml`.

### Upgrade gh-aw (`gh aw upgrade`)

`gh aw upgrade` migrates `.md` via codemods, recompiles, and scaffolds new files.
It can leave a `.md` that no longer compiles when the schema moved (codemods don't
catch everything). Worked example from the MIF upgrade:

- `network.firewall: true` → `sandbox.agent: awf`
- `tools.memory: cache-memory` → `tools.cache-memory:` (memory moved under
  `sandbox/agent`; `cache-memory` became a first-class tool)

Steps:

1. Run `gh aw upgrade`. If it reports `N workflow(s) failed to compile`, read the
   error — it names the file:line and the invalid property.
2. Fix the **`.md`** (never the `.lock.yml`), then `gh aw compile` until 0 errors.
3. Review the scaffolded files (`agentics-maintenance.yml`, `copilot-setup-steps.yml`,
   `.github/{agents,skills,aw}`) — these are **new CI workflows and config**, not
   just a fix. Treat the upgrade as a reviewable PR.
4. Run the [validation checklist](#pre-push-validation-checklist).

### Refresh stale pins / `actions-lock.json` drift

Symptom: `actions-lock.json` lists container/action pins that no longer match the
recompiled `.lock.yml` manifest (e.g. old-stack `gh-aw-firewall 0.20.2` /
`mcpg v0.1.4` / `github-mcp-server v0.30.3` while the manifest uses `0.27.11` /
`v0.3.30` / `v1.4.0`).

Fix:

```sh
gh aw compile --force-refresh-action-pins
```

This clears the cache and re-resolves. On the current stack it **drops the stale
`containers` block from `actions-lock.json`** — that's expected: the compiled
`.lock.yml` now carries the container digests inline (`…:tag@sha256:…`), so the
separate container cache is redundant. Verify the manifest is digest-pinned:

```sh
grep -oE 'ghcr\.io/[^"@ ]+@sha256:[0-9a-f]+' .github/workflows/<name>.lock.yml | sort -u
```

Commit the refreshed `actions-lock.json` (and any `.lock.yml` changes).

### Drift: Dependabot vs recompile (#101)

`.lock.yml` carries the gh-aw setup action pinned by SHA with an inline version
comment (`uses: github/gh-aw/actions/setup@<sha> # vX.Y.Z`). Dependabot bumps the
**SHA** without recompiling, so the comment and the `compiler_version` metadata
**drift** from the SHA they label. A clean recompile (`gh aw compile`) resyncs the
comment, metadata, and scattered schedule — that is the authoritative fix. (To
*prevent* recurrence, the maintainer decides between ignoring the generated
`*.lock.yml` path in `dependabot.yml` or a scheduled recompile job — out of scope
for a single recompile.)

### Keeping standalone linters green on generated output

The org's **standalone** actionlint/checkov don't know gh-aw's extensions, so they
flag generator output that `gh aw lint`/`gh aw validate` accept. Do **not** hand-edit
the generated files — scope-ignore instead.

#### actionlint — `.github/actionlint.yaml`

actionlint auto-discovers `.github/actionlint.yaml`. Scope each known generator
quirk to its generated file (matches messages by regex). Seen so far:

```yaml
self-hosted-runner:
  labels: [ubuntu-slim]
paths:
  .github/workflows/<name>.lock.yml:
    ignore:
      - 'property "pre_activation" is not defined'
      - 'shellcheck reported issue in this script'
      - 'unexpected key "queue" for "concurrency" section'
  .github/workflows/agentics-maintenance.yml:
    ignore:
      - 'string should not be empty'
```

Verify: `actionlint` (repo root) exits 0.

#### checkov — soft-fail, dismiss false positives

The org's checkov reusable runs `--soft-fail`: it reports to code-scanning but does
**not** fail the job, so its findings are advisory (non-blocking). Handle gh-aw
generated findings case by case:

- **`CKV_GHA_7`** ("workflow_dispatch inputs MUST be empty") on the generated
  `agentics-maintenance.yml`: its `operation`/`run_url` inputs are intentional and
  the file is a maintenance/ops workflow, not a build. **Dismiss the code-scanning
  alert as a false positive** (it's generated; you can't hand-edit it).
- **`CKV2_GHA_1`** ("top-level permissions not write-all") on the **scaffolded**
  `copilot-setup-steps.yml` (hand-editable): add a top-level least-privilege block:
  ```yaml
  permissions:
    contents: read
  ```

#### pin-check (scaffold files)

`pin-check` requires every `uses:` to be a 40-hex SHA. `gh aw compile` SHA-pins the
`.lock.yml`, but **scaffolded** files (`copilot-setup-steps.yml`) may carry tag
refs (`actions/checkout@v6`). Pin them by resolving the real SHA — never guess:

```sh
gh api repos/actions/checkout/commits/v6 --jq .sha        # -> df4cb1c… (v6.0.3)
# uses: actions/checkout@df4cb1c069e1874edd31b4311f1884172cec0e10 # v6.0.3
```

## Pre-push validation checklist

Run from the repo root before pushing any gh-aw change:

```sh
gh aw validate                 # compile + linters, no emit -> 0 errors
gh aw compile && git status    # tree must be clean (no stale .lock.yml)
actionlint                     # exits 0 with .github/actionlint.yaml
python3 -c 'import json; json.load(open(".github/aw/actions-lock.json"))'   # valid
```

Note: the org's `schema-check` / spec gates only run on PRs targeting
`main`/`release/**`/`develop/**`. A PR based on a feature branch (a stacked PR) does
**not** trigger them — run them locally too (see the repo's `CLAUDE.local.md`).

## Gotchas

- **Never hand-edit `.lock.yml` or `# DO NOT EDIT` files** — recompile overwrites them.
- **`gh aw lint` passing ≠ standalone actionlint passing.** gh-aw tolerates its own
  extensions; the org check does not. Both must be addressed (the latter via
  `.github/actionlint.yaml`).
- **The upgrade adds real CI workflows** (`agentics-maintenance.yml`,
  `copilot-setup-steps.yml`) that run in Actions — review them, don't rubber-stamp.
- **Container digests live in the `.lock.yml` manifest**, not (on the current stack)
  in `actions-lock.json`. After `--force-refresh-action-pins`, an empty `containers`
  block is correct.
